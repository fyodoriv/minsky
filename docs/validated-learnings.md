- `openhands-integration-shipped-2026-05-24` — recorded 2026-05-24 (PR #782): the operator's "complete OpenHands integration today, aggressively replace minsky parts with openhands but with minsky principles, make openhands default" directive landed end-to-end in one PR. On 2026-05-24 the `@minsky/agent-runtime-openhands` package shipped: a Python shim wrapping OpenHands SDK v1.19.1 in-process (~245 LOC) + a pure-TS spawn-config builder (~125 LOC) + 20 unit tests + 6 Python-shim boundary tests. The shim is explicitly throwaway — when OpenHands' Agent Canvas Initiative CLI ships on 2026-06-01, the shim is replaced one-for-one with a direct `openhands solve --task-file X` invocation and the TS adapter shape stays the same. Default flipped at the same time: `resolveCloudAgent`'s default went from `"claude"` to `"openhands"`; the audit-matrix lint's date-aware self-flipping branch was removed (post-integration steady state asserted instead). The KEY lesson: the substrate-first pattern (AGENT_MATRIX with `pendingExternalDep` discriminator landed three weeks ahead of the runtime) made TODAY's integration a 5-file mechanical flip, not a debugging session — the matrix forced the wire-shape contract to be designed in isolation, so on integration day the only question was "does this Python SDK actually run?" (yes, verified locally with openhands-ai 1.19.1). Replicate the substrate-first pattern for the next pluggable interface: skill loader → OpenHands AgentSkills-spec.

The KEY lesson: **strategic stances that name when they expire** are how a strong-opinions repo evolves without ossifying — story 015 names the *expiry condition* (90% stability) and the *unlock action* (cloud A/B), so the stance is auditable and the unlock isn't subject to negotiation; story 014 names the *invariant* and a *chaos test* so future launcher-coupling can't sneak in silently. Replicate this shape: every strategic-stance user story should declare (a) the invariant in one line, (b) the expiry condition / chaos test that catches violation, (c) the unlock action with a follow-up task ID.
