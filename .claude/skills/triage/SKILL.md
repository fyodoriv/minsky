# Triage

Move TASKS.md entries through a small state machine of task states.

Every AI-generated comment posted to a GitHub issue or PR during triage **must** start with:

```
> *This was generated by AI during triage.*
```

When editing TASKS.md directly, add an HTML comment on the same line noting the triage action:
```
<!-- triage: moved to blocked, reason: needs operator decision on X -->
```

## Task States in Minsky

Minsky's task states are encoded in TASKS.md markers and the `**Status**` field:

| State | Marker / Field | Meaning |
|---|---|---|
| `needs-triage` | `- [ ]` with no `**Status**` | New task, not yet evaluated |
| `needs-info` | `**Status**: blocked` + comment | Waiting on operator for missing info |
| `ready-for-agent` | `- [ ]` in correct P-section, all fields present | Fully specified, safe for daemon pickup |
| `in-progress` | `(@<agent-id>)` suffix OR `**Status**: in-progress` | Claimed by a worker |
| `wontfix` | Block removed (history in git log) | Will not be actioned |
| `shipped` | `[x]` OR `**Status**: shipped` | Done; auto-removal via daemon-task-rotation |

Two **category** tags (set in `**Tags**`):
- `bug` — something is broken
- `enhancement` — new feature or improvement

Every triaged task should carry exactly one category tag and one state marker. Flag contradictions before acting.

## Invocation

The operator invokes `/triage` and describes what they want in natural language. Examples:

- "Show me anything that needs my attention"
- "Triage `daemon-stuck-pr-rebase-watchdog`"
- "Mark `foo-task` as blocked — needs operator decision on auth model"
- "What's ready for agents to pick up?"
- "Write an agent brief for `daemon-pre-pr-lint-gate`"

## Show what needs attention

Scan TASKS.md and present three buckets, oldest first within each:

1. **Unlabeled / no-fields** — task line with no metadata block at all.
2. **`needs-triage` candidates** — `- [ ]` tasks with incomplete required fields (missing `**Hypothesis**`, `**Measurement**`, `**Verification**`, or `**Tags**`).
3. **`blocked` with recent operator activity** — tasks with `**Status**: blocked` where the operator has since commented or the blocker condition appears resolved.

Show counts and one-line summary per task. Let the operator pick.

## Triage a specific task

1. **Gather context.** Read the full task block from TASKS.md (all fields). Read the files listed in `**Touches**` / `**Files**`. Check open and recently merged PRs for this task ID (`gh pr list --search "<task-id>"`). Check `.out-of-scope/*.md` for prior rejection of the same concept.

2. **Recommend state + priority.** Tell the operator the recommended state, priority section (P0–P3), and any missing fields, with reasoning. Wait for direction.

3. **Reproduce (bugs only).** Before writing any brief, attempt reproduction: read the `**Hypothesis**` / `**Details**` fields, trace relevant code, run tests. Report: confirmed repro (code path), failed repro, or insufficient detail (`needs-info` signal).

4. **Fill missing fields (if needed).** If the task needs completing, work through the fields one at a time using `/grill-task` style: one question per turn, recommended answer, wait for operator response.

5. **Apply the outcome:**

   - `ready-for-agent` — ensure all required fields present (ID, Tags, Hypothesis, Measurement, Verification, Anchor, Files, Touches). Optionally append an **Agent Brief** block (see template below).
   - `needs-info` → set `**Status**: blocked` + append a `<!-- triage: blocked, needs: <exact question> -->` comment.
   - `wontfix` (bug) — remove block + commit with message `chore(tasks): wontfix <task-id> — <reason>`.
   - `wontfix` (enhancement) — write `.out-of-scope/<concept>.md` (see Out-of-Scope section below), then remove block.
   - Priority change — move block to correct P-section; ensure `**Tags**` contains the matching `p0`/`p1`/`p2`/`p3` tag.

## Quick state override

If the operator says "mark `foo-task` ready-for-agent", trust them and apply the state directly. Confirm what you're about to change (fields, section move, comment), then act. Skip grilling. If moving to ready-for-agent without a grill session, ask whether they want an agent brief written.

## Required task fields for ready-for-agent

A task is `ready-for-agent` only when ALL of these fields are present and non-empty:

- `**ID**` — matches the task slug in the list item
- `**Tags**` — contains at least one of `bug`/`enhancement` AND at least one of `p0`/`p1`/`p2`/`p3`
- `**Hypothesis**` — falsifiable pre-registration (rule #9)
- `**Measurement**` — exact runnable command
- `**Verification**` — acceptance criteria checkboxes
- `**Anchor**` — at least one cited source
- `**Files**` + `**Touches**` — lists affected files
- `**Estimate**` — size in hours or days

## Agent Brief template

Append as a fenced block under the task entry when moving to `ready-for-agent`:

```markdown
**Agent Brief:**

**Category:** bug / enhancement
**Summary:** one-line description of what needs to happen

**Current behavior:**
What happens now.

**Desired behavior:**
What should happen after the agent's work is complete. Specific about edge cases.

**Key interfaces:**
- `TypeName` / function signature / config shape that needs to change

**Acceptance criteria:**
- [ ] Specific, testable criterion 1
- [ ] Specific, testable criterion 2

**Out of scope:**
- Adjacent concern that should NOT be changed
```

Agent briefs must be **behavioral, not procedural** — describe what the system should do, not how to implement it. No file paths (they go stale). No line numbers. Reference types, interfaces, and behavioral contracts instead.

## Out-of-Scope knowledge base

When an enhancement is rejected as wontfix, write `.out-of-scope/<concept>.md`:

```markdown
# <Concept Name>

This project does not support <feature>.

## Why this is out of scope

<Substantive reason referencing architecture, scope, or strategy.>

## Prior requests

- TASKS.md `<task-id>` — "<task tagline>" (removed <date>)
```

During triage Step 1, read all `.out-of-scope/*.md` files and surface matches:
"This is similar to `.out-of-scope/local-graphql-api.md` — rejected before because [reason]. Still out of scope?"

Matching is by concept similarity, not keyword. Maintainer confirms or reconsiders.

## Resuming a previous triage session

If the task already has triage notes or a `**Status**` field, read them, check whether the blocking condition is resolved, and present an updated picture before continuing. Do not re-ask resolved questions.
